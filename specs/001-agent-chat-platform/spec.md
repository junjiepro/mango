# Feature Specification: Mango - 智能Agent对话平台

**Feature Branch**: `001-agent-chat-platform`
**Created**: 2025-11-24
**Status**: Draft
**Input**: User description: "帮我构建一个应用，能帮助用户完成各式各样的任务。核心功能是一个聊天窗口，与agent对话，agent能使用工具完成任务，支持多模态输入输出，支持mcp、A2A、ACP等协议。就算用户离开窗口，agent也能在后台继续任务它的任务。agent可以在处理任务的过程中持续学习，下一次能更好地完成任务。在对话中，agent能构建一些小应用，这些小应用可以被动地被agent或者用户使用，比如todolist应用中对ToDo的增删改查，也可以应用自己主动触发，比如todolist主动发出消息，提醒今天的待办事项。另外，小应用是可以分享给其他用户使用的。支持桌面端，移动端和web端，当前阶段可主要关注web端。另外，需要开发一个cli工具，便于用户把其本地的mcp或者ACP接入到应用中。UI风格要求简洁明了，色彩不宜过多，对于不同内容和场景应该有合适UI，比如，对于会话中只有一个图片或者文件，可以简单地显示，若有多个图片或者文件，应该有一个目录结构供用户方便访问。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 与Agent进行对话完成任务 (Priority: P1)

用户打开Web应用，在聊天窗口中输入任务描述（文本、图片或文件），Agent理解用户意图并使用可用工具逐步完成任务，过程中实时向用户反馈进度和结果。

**Why this priority**: 这是平台的核心价值主张——通过自然语言对话让Agent帮助用户完成任务。没有这个功能，其他所有功能都失去意义。

**Independent Test**: 可以通过发送一条简单的文本指令（如"帮我查询今天的天气"）并观察Agent的响应来完整测试，交付用户通过对话获得帮助的核心价值。

**Acceptance Scenarios**:

1. **Given** 用户已登录且处于聊天界面, **When** 用户输入文本消息"帮我总结这篇文章"并上传一个PDF文件, **Then** Agent接收消息并开始处理，显示处理进度，最终返回文章摘要
2. **Given** Agent正在执行任务, **When** 用户关闭浏览器窗口后重新打开, **Then** 用户能看到任务仍在进行或已完成，并可查看完整的任务历史记录
3. **Given** 用户发送包含多张图片的消息, **When** Agent完成分析, **Then** 界面以目录结构形式展示分析结果，用户可方便地浏览每张图片的分析详情

---

### User Story 2 - 使用和管理小应用 (Priority: P2)

用户在对话过程中，Agent识别到可以创建小应用来更好地服务用户需求（如待办事项管理），创建后用户可以直接在对话中使用该小应用，小应用也可以主动发送提醒消息。

**Why this priority**: 小应用扩展了平台的能力边界，让Agent不仅能完成一次性任务，还能提供持续性的服务价值。这是产品差异化的关键功能。

**Independent Test**: 可以通过让Agent创建一个简单的待办事项小应用，添加一条待办，设置提醒时间，验证提醒是否按时触发来完整测试。

**Acceptance Scenarios**:

1. **Given** 用户说"帮我创建一个待办事项列表", **When** Agent完成创建, **Then** 用户在对话中看到一个可交互的待办事项组件，可直接添加、编辑、删除待办项
2. **Given** 用户在待办事项小应用中设置了"明天上午9点提醒我开会", **When** 到达指定时间, **Then** 小应用主动在对话中发送提醒消息，即使用户当时未在线也会在上线后收到
3. **Given** 用户拥有一个有用的小应用, **When** 用户选择分享该小应用, **Then** 生成一个可分享的链接，其他用户可通过链接将该小应用添加到自己的工作空间

---

### User Story 3 - 通过CLI工具接入本地MCP/ACP服务 (Priority: P3)

技术用户通过CLI工具在本地启动设备服务，该服务通过 Cloudflare Tunnel 暴露到公网，用户可以将设备与 Mango Web 账户绑定，从而让平台上的 Agent 调用本地的 MCP/ACP 服务，扩展 Agent 的工具能力。

**Why this priority**: 这使得平台具有可扩展性，允许用户集成自己的工具和服务，大大增加平台的应用场景，但需要在核心功能稳定后实现。

**Independent Test**: 可以通过安装CLI工具、启动设备服务、完成设备绑定、配置本地MCP服务、在对话中让Agent调用该服务来完整测试。

**Acceptance Scenarios**:

1. **Given** 用户已安装CLI工具, **When** 用户首次运行CLI工具启动设备服务, **Then** 系统自动生成 device_secret 并打印，启动本地服务(包含 /health、/mcp、/acp、/setting、/bind 端点)，创建 Cloudflare Tunnel，并自动打开浏览器跳转到绑定页面
2. **Given** 用户访问绑定页面, **When** 用户输入 device_secret 并确认绑定, **Then** 设备与用户账户建立绑定关系，tunnel URL 保存到远程数据库，用户可以看到设备在线状态
3. **Given** 设备已成功绑定, **When** 用户通过 /setting 端点配置多个本地 MCP 服务, **Then** 配置保存成功，设备服务通过 /mcp 端点提供 streamable HTTP MCP 服务，可访问这些 MCP 服务的工具和资源
4. **Given** 本地MCP服务已配置, **When** 用户在 Mango Web 对话中请求使用该服务提供的功能, **Then** Agent 通过 tunnel 调用设备服务，设备服务转发请求到本地 MCP 服务，完成任务并返回结果
5. **Given** 用户需要在不同场景使用不同配置, **When** 用户创建多个设备绑定, **Then** 每个绑定拥有独立的设置和资源隔离，用户可以在不同绑定间切换
6. **Given** 用户启动CLI工具时指定参数, **When** 用户使用 --ignore-open-bind-url 参数, **Then** 系统不自动打开绑定页面，仅在控制台显示绑定 URL
7. **Given** 设备服务运行中, **When** 本地 MCP 服务连接断开或返回错误, **Then** 设备服务向 Agent 返回明确的错误信息，Agent 向用户说明服务不可用并建议检查本地服务状态

---

### User Story 4 - Agent持续学习与改进 (Priority: P4)

Agent在与用户交互过程中学习用户偏好和反馈，在后续对话中能够提供更个性化、更准确的服务。

**Why this priority**: 持续学习让Agent越用越好用，提高用户粘性和满意度，但这是在基础功能完善后的增强功能。

**Independent Test**: 可以通过多次让Agent执行同类任务，对结果进行正向/负向反馈，观察Agent后续执行同类任务时是否有改进来测试。

**Acceptance Scenarios**:

1. **Given** 用户多次请求Agent以特定格式输出内容, **When** 用户下次发起类似请求, **Then** Agent自动采用用户偏好的格式，无需用户再次说明
2. **Given** Agent完成一项任务, **When** 用户对结果表示不满意并说明原因, **Then** Agent记录反馈，下次遇到类似任务时避免同样的问题
3. **Given** 用户查看Agent的学习记录, **When** 用户选择删除某些学习数据, **Then** 系统清除相关数据，Agent不再基于这些数据做决策

---

### User Story 5 - 富交互界面与工作区 (Priority: P5)

Agent能够生成丰富的用户交互界面（A2UI），会话中出现的资源（文件、链接、小应用等）被自动收集到对话框下方便于快速访问；用户可以展开/收起工作区，工作区提供多标签页界面用于查看资源、设备状态、文件浏览（Monaco Editor）、多终端操作等功能。

**Why this priority**: 这是用户体验的重大提升，让用户能够在对话中获得更丰富的交互体验，同时通过工作区提供专业级的开发和管理能力，大幅提升平台的实用性和专业性。

**Independent Test**: 可以通过让Agent生成一个包含交互组件的界面，上传多个文件验证资源收集，展开工作区查看设备文件系统并在终端执行命令来完整测试。

**Acceptance Scenarios**:

1. **Given** Agent需要收集用户输入, **When** Agent使用A2UI生成表单界面, **Then** 用户在对话中看到可交互的表单组件（输入框、下拉选择、按钮等），填写后数据传回Agent
2. **Given** Agent返回数据可视化结果, **When** Agent使用A2UI生成图表, **Then** 用户在对话中看到交互式图表（可缩放、筛选、导出）
3. **Given** 对话中出现文件、链接或小应用, **When** 资源嗅探功能检测到这些内容, **Then** 资源自动收集到对话框下方的资源面板，按类型分组展示，用户可快速点击访问
4. **Given** 用户需要查看更多信息, **When** 用户点击展开工作区按钮, **Then** 工作区从底部或侧边展开，对话区自动调整尺寸，用户可拖动分隔条调整两者比例
5. **Given** 工作区已展开, **When** 用户查看工作区, **Then** 工作区显示多个标签页（资源、设备、文件浏览器、终端等），用户可在标签间切换
6. **Given** 用户在工作区的"资源"标签, **When** 用户浏览, **Then** 显示会话中所有收集的资源，支持按类型筛选、搜索、预览和下载
7. **Given** 用户在工作区的"设备"标签且已绑定设备, **When** 用户查看, **Then** 显示已选择设备的在线状态、系统信息、配置的MCP/ACP服务列表
8. **Given** 用户在工作区的"文件浏览器"标签, **When** 用户选择已绑定的设备, **Then** 使用Monaco Editor显示设备的文件目录树，用户可浏览目录、打开文件查看/编辑内容
9. **Given** 用户在工作区的"终端"标签, **When** 用户创建新终端, **Then** 打开一个连接到选定设备的终端会话，用户可执行命令、查看输出，支持同时打开多个终端标签
10. **Given** 用户在小屏幕设备上使用, **When** 用户展开工作区, **Then** 工作区以全屏模式覆盖对话区，提供"返回对话"按钮，确保小屏幕上的良好体验

---

### User Story 6 - 多语言国际化支持 (Priority: P6)

用户可以选择自己偏好的界面语言，系统界面、提示信息、错误消息等均以用户选择的语言显示；同时Agent能够理解和回复用户使用的任何语言。

**Why this priority**: 国际化支持让平台能服务全球用户，扩大用户群体，但核心功能稳定后再实现更为合理。

**Independent Test**: 可以通过切换界面语言到英文，验证所有界面元素是否正确显示英文；然后用中文与Agent对话，验证Agent能正常理解和回复。

**Acceptance Scenarios**:

1. **Given** 用户首次访问平台, **When** 系统检测到用户浏览器语言为英文, **Then** 界面默认显示为英文，用户可在设置中更改
2. **Given** 用户在设置中将语言切换为日语, **When** 切换完成, **Then** 所有界面元素（菜单、按钮、提示文字、错误消息）立即切换为日语显示
3. **Given** 界面语言设置为英文, **When** 用户用中文向Agent发送消息, **Then** Agent用中文回复用户，界面元素仍保持英文显示
4. **Given** 用户分享小应用给使用不同语言的用户, **When** 接收方打开小应用, **Then** 小应用界面按接收方的语言偏好显示

---

### Edge Cases

- 当网络连接中断时，系统保存用户输入的草稿，网络恢复后提示用户是否继续发送
- 当用户选择的语言暂无完整翻译时，系统显示已翻译部分，未翻译部分回退到默认语言（英文）并标注
- 当Agent长时间无响应（超过2分钟）时，系统显示超时提示并提供重试或取消选项
- 当用户同时在多个设备登录时，对话和任务状态实时同步，任务只执行一次
- 当小应用尝试发送提醒但用户已离线超过24小时时，系统聚合多条提醒为摘要通知
- 当上传的文件超过大小限制时，系统给出明确提示并建议分割或压缩
- 当本地MCP/ACP服务返回错误格式数据时，Agent优雅处理并向用户说明问题

## Requirements *(mandatory)*

### Functional Requirements

**核心对话功能**
- **FR-001**: 系统必须提供实时对话界面，支持用户与Agent进行自然语言交互
- **FR-002**: 系统必须支持多模态输入，包括文本、图片、文件上传
- **FR-003**: 系统必须支持多模态输出，Agent可返回文本、图片、文件、结构化数据等
- **FR-004**: 系统必须在用户离开界面后继续执行Agent任务，并在用户返回时展示进度和结果
- **FR-005**: 系统必须保存完整的对话历史，用户可随时查看和搜索

**协议支持**
- **FR-006**: 系统必须支持MCP（Model Context Protocol）协议，允许Agent调用符合该协议的工具服务
- **FR-007**: 系统必须支持A2A（Agent-to-Agent）协议，允许Agent与其他Agent协作 *(Phase 2实现，当前版本仅完成架构预留)*
- **FR-008**: 系统必须支持ACP（Agent Client Protocol, The Agent Client Protocol standardizes communication between code editors/IDEs, and coding agents (programs that use generative AI to autonomously modify code).）协议，允许Agent接入符合该协议的coding agents

**小应用功能**
- **FR-009**: 系统必须允许Agent在对话过程中创建小应用
- **FR-010**: 小应用必须支持被动调用模式，用户或Agent可通过指令使用小应用功能
- **FR-011**: 小应用必须支持主动触发模式，可根据预设条件主动发送消息
- **FR-012**: 系统必须支持小应用的分享功能，用户可将小应用分享给其他用户
- **FR-013**: 系统必须提供小应用的权限管理，用户可控制小应用的访问范围

**学习与改进**
- **FR-014**: 系统必须记录用户对Agent输出的反馈（满意/不满意及原因）
- **FR-015**: Agent必须能够基于历史反馈优化后续任务执行
- **FR-016**: 系统必须允许用户查看和管理Agent的学习数据

**CLI工具与设备服务**
- **FR-017**: 必须提供CLI工具用于启动本地设备服务，支持命令行参数配置（ignore_open_bind_url、app_url、supabase_url、supabase_anon_key、device_secret）
- **FR-018**: CLI工具启动时必须自动生成 device_secret（如不存在）并打印到控制台
- **FR-019**: 设备服务必须提供以下 HTTP 端点：/health（健康检查）、/mcp（MCP协议服务）、/acp（ACP协议服务）、/setting（配置管理）、/bind（设备绑定）
- **FR-020**: 设备服务启动后必须自动创建 Cloudflare Tunnel 将服务暴露到公网
- **FR-021**: 设备服务必须支持设备与用户的多对多绑定关系，每个绑定拥有独立的配置和资源隔离
- **FR-022**: 设备绑定时必须验证 device_secret，确保只有授权用户可以绑定设备
- **FR-023**: 系统必须将 Cloudflare Tunnel URL 保存到远程数据库，用于 Agent 与设备的通信
- **FR-024**: 设备服务必须支持通过 /setting 端点配置多个本地 MCP/ACP 服务
- **FR-025**: 设备服务必须通过 /mcp 端点提供 streamable HTTP MCP 服务，代理访问本地配置的 MCP 服务
- **FR-026**: 未绑定的设备首次启动时必须自动打开浏览器跳转到绑定页面（除非指定 ignore_open_bind_url 参数）
- **FR-027**: 设备服务必须提供安全的认证机制，防止未授权的服务接入和调用

**用户界面**
- **FR-028**: Web界面必须采用简洁明了的设计风格，色彩使用克制
- **FR-029**: 系统必须根据内容类型和数量自适应选择展示方式
- **FR-030**: 多文件场景必须提供目录结构视图，便于用户浏览和访问

**富交互界面（A2UI）**
- **FR-038**: Agent必须能够生成富交互界面组件（表单、按钮、下拉选择、图表等），嵌入到对话消息中
- **FR-039**: A2UI组件必须支持用户交互，用户操作数据必须能够传回Agent进行处理
- **FR-040**: A2UI必须支持常见的数据可视化组件（折线图、柱状图、饼图等），支持交互式操作（缩放、筛选、导出）
- **FR-041**: A2UI组件必须遵循统一的设计系统，与整体界面风格保持一致

**资源嗅探与管理**
- **FR-042**: 系统必须自动检测对话中出现的资源（文件、链接、小应用等）
- **FR-043**: 检测到的资源必须自动收集到对话框下方的资源面板中
- **FR-044**: 资源面板必须按类型分组展示资源（文件、链接、小应用等）
- **FR-045**: 用户必须能够在资源面板中快速访问、预览、下载资源
- **FR-046**: 资源面板必须支持搜索和筛选功能

**工作区功能**
- **FR-047**: 系统必须提供可展开/收起的工作区组件
- **FR-048**: 工作区展开时，对话区必须自动调整尺寸，用户可拖动分隔条调整两者比例
- **FR-049**: 工作区必须提供多标签页界面（资源、设备、文件浏览器、终端等）
- **FR-050**: 工作区的"资源"标签必须显示会话中所有收集的资源，支持按类型筛选、搜索、预览和下载
- **FR-051**: 工作区的"设备"标签必须显示已选择设备的在线状态、系统信息、配置的MCP/ACP服务列表
- **FR-052**: 工作区的"文件浏览器"标签必须使用Monaco Editor显示设备的文件目录树
- **FR-053**: 文件浏览器必须支持浏览目录、打开文件查看内容、编辑文件内容
- **FR-054**: 工作区的"终端"标签必须支持创建多个终端会话，连接到选定的设备
- **FR-055**: 终端必须支持执行命令、查看输出、支持常见的终端快捷键
- **FR-056**: 在小屏幕设备上，工作区展开时必须以全屏模式覆盖对话区，提供"返回对话"按钮

**用户管理**
- **FR-031**: 系统必须支持用户注册和登录
- **FR-032**: 系统必须支持多设备同时登录，数据实时同步

**国际化支持**
- **FR-033**: 系统必须支持多语言界面，初始支持中文、英文，架构可扩展支持更多语言
- **FR-034**: 系统必须能够根据用户浏览器语言自动检测并设置默认界面语言
- **FR-035**: 用户必须能够在设置中手动切换界面语言，切换后立即生效无需刷新
- **FR-036**: Agent必须能够识别用户消息的语言，并以相同语言进行回复
- **FR-037**: 小应用的界面文本必须支持国际化，按使用者的语言偏好显示

### Key Entities

- **User（用户）**: 使用平台的终端用户，拥有账户、偏好设置（含语言偏好）、对话历史
- **Conversation（对话）**: 用户与Agent之间的会话，包含多条消息，有明确的开始时间，可持续进行
- **Message（消息）**: 对话中的单条信息，可由用户或Agent发送，支持多模态内容（文本、图片、文件、结构化数据、A2UI组件）
- **Agent**: 智能助手实体，具备使用工具、创建小应用、持续学习、生成A2UI界面的能力
- **Task（任务）**: Agent执行的具体工作单元，有状态（进行中/已完成/失败）、进度、结果
- **MiniApp（小应用）**: Agent创建的可复用功能模块，具有数据存储、用户交互、主动触发能力，可分享
- **Tool（工具）**: Agent可调用的外部能力，通过MCP/A2A/ACP协议接入
- **Device（设备）**: 运行CLI工具的本地设备，拥有唯一的 device_secret，通过 Cloudflare Tunnel 暴露服务到公网
- **DeviceBinding（设备绑定）**: 设备与用户之间的绑定关系，多对多关系，每个绑定拥有独立的配置、tunnel URL、在线状态
- **DeviceService（设备服务）**: 运行在本地设备上的 HTTP 服务，提供 /health、/mcp、/acp、/setting、/bind 端点
- **MCPService（MCP服务）**: 配置在设备服务中的本地 MCP 协议服务，提供工具和资源供 Agent 调用
- **ACPService（ACP服务）**: 配置在设备服务中的本地 ACP 协议服务，提供 coding agent 能力
- **CloudflareTunnel（Cloudflare隧道）**: 由设备服务自动创建的临时隧道，将本地服务暴露到公网，URL 保存在数据库中
- **LearningRecord（学习记录）**: Agent从用户反馈中提取的经验，用于优化后续行为
- **Attachment（附件）**: 消息中的媒体文件或文档，有类型、大小、预览信息
- **A2UIComponent（A2UI组件）**: Agent生成的富交互界面组件，可嵌入消息中，支持表单、图表、按钮等交互元素
- **Resource（资源）**: 对话中出现的可访问内容，包括文件、链接、小应用等，被资源嗅探功能自动收集
- **ResourcePanel（资源面板）**: 显示在对话框下方的面板，展示收集到的资源，支持分类、搜索、快速访问
- **Workspace（工作区）**: 可展开/收起的多标签页界面，提供资源管理、设备监控、文件浏览、终端操作等功能
- **WorkspaceTab（工作区标签）**: 工作区中的单个标签页，如资源标签、设备标签、文件浏览器标签、终端标签
- **FileExplorer（文件浏览器）**: 工作区中的文件管理组件，使用Monaco Editor显示设备文件系统，支持浏览、查看、编辑
- **Terminal（终端）**: 工作区中的终端组件，连接到选定设备，支持命令执行、输出查看、多终端会话

## Assumptions

- 用户有稳定的网络连接（移动网络或WiFi）
- 用户使用现代浏览器（Chrome、Firefox、Safari、Edge的最近两个主要版本）
- 后端服务具备足够的计算资源支持Agent任务执行
- MCP、A2A、ACP协议的具体规范可获取并可实现
- 用户愿意提供反馈以帮助Agent学习改进

## Scope Boundaries

**包含在范围内**:
- Web端完整功能实现
- CLI工具开发
- MCP/A2A/ACP协议基础集成
- 核心小应用能力（创建、使用、分享）
- 基础学习能力
- 国际化支持（中文、英文）

**不在当前范围内**:
- 桌面端原生应用（未来迭代）
- 移动端原生应用（未来迭代）
- A2A协议完整实现（Phase 2，当前版本仅架构预留）
- 小应用的商业化分发市场
- 企业级多租户支持
- 更多语言支持（日语、韩语等，未来按需扩展）

## Technical Constraints

- **Web前端框架**: Next.js (React全栈框架，支持SSR/SSG)
- **后端运行时**: Node.js + Deno + TypeScript (全栈统一语言生态)
- **数据库**: Supabase (PostgreSQL托管服务，支持JSON/JSONB，全文搜索)
- **用户认证**: Supabase Auth (邮箱密码+OAuth社交登录+魔法链接)
- **文件存储**: Supabase Storage (统一生态，支持公开/私有bucket)
